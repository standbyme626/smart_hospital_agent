smart_hospital_agent
四大核心模块拆分实施方案
聚焦第一阶段：config / diagnosis / retriever / chat-stream
目标	先把最臃肿、最影响迭代效率的四块拆开，稳定协议，再逐步收敛复杂度。
拆分对象	1) core/config.py  2) core/graph/sub_graphs/diagnosis.py  3) rag/retriever.py  4) api/v1/endpoints/chat.py + services/chat_service.py
原则	接口先固化、行为先等价、灰度可回退、每步都能测试、不要同时重写业务逻辑。
预期收益	降低单文件复杂度、减少跨层耦合、让路由/RAG/流式链路可测试、为后续专家组与21科室收口创造前提。

一、为什么先拆这四块
这四块是当前系统的复杂度汇聚点：配置决定行为边界，诊断子图承载了路由与状态同步，检索模块承载了策略聚合，聊天流式链路又把 HTTP、SSE、图事件和用户体验揉在一起。先解决这四块，后续专家组、科室扩容、治理与测试才能真正落地。
本方案采用“先抽协议、再搬逻辑、最后收敛实现”的思路，不追求一步到位重写，而是用 3 个阶段把模块从超大文件拆成有边界的小模块。
二、总体实施策略
阶段	核心目标	动作	交付物
Phase 0	先立边界	冻结现有行为，补充接口清单、关键状态字段、主要调用链与 smoke case。	现状清单、接口文档、回归样本
Phase 1	拆壳不改心	创建新目录和 façade，旧逻辑先原样搬迁或包裹，确保 import 路径可兼容。	新目录、新入口、兼容层
Phase 2	按职责下沉	把路由、状态、缓存、重排、SSE 适配等逻辑逐步下沉到子模块。	职责清晰的子模块
Phase 3	验证与收口	补 contract test、链路 trace、回退开关与迁移文档，移除废弃调用。	测试、指标、迁移说明

三、模块一：拆分 core/config.py
当前痛点
• 一个 Settings 承担了数据库、缓存、模型、RAG、设备、特性开关等多类职责，新增配置会继续放大耦合。
• 配置既有稳定项也有实验开关，排障时很难判断“当前到底哪些值在生效”。
• 配置值和行为逻辑纠缠，例如 fallback、key 候选、路径派生等都混在一个对象里。
目标目录与职责
模块/文件	职责
core/settings/base.py	应用名、环境、API 前缀、基础路径、公共枚举。
core/settings/storage.py	数据库、Redis、Milvus、文件路径、会话持久化相关配置。
core/settings/model.py	主模型、本地模型、embedding、provider、设备与回退策略。
core/settings/rag.py	召回、重排、缓存、pure mode profile、阈值与开关。
core/settings/observability.py	日志、metrics、trace、调试输出、采样率。
core/settings/feature_flags.py	只允许短生命周期实验开关，明确默认值和下线日期。
core/settings/__init__.py	统一组装入口，对外继续暴露 settings 对象。

拆分步骤
1. 先从现有 config.py 提取字段清单，按“稳定配置/实验开关”打标签；不要一边拆一边改值。
2. 创建 settings package 与新的 SettingsFacade，对外仍提供 `from app.core.config import settings` 的兼容入口。
3. 把纯值字段先迁出；把带行为的派生逻辑改为独立 helper，例如 model_resolver、path_builder、key_selector。
4. 为每个 settings 子模块增加 `summary()` 或 `effective_snapshot()`，便于 `/health`、日志和调试输出当前生效配置。
5. 第二轮再收紧命名：统一 prefix、统一布尔开关命名、把 pure mode 变成 profile 而不是多布尔拼装。
关键接口与保留兼容
• 保留 `settings = get_settings()` 统一入口，避免业务代码直接实例化子配置。
• 保留旧字段别名映射表，至少跨一个版本周期。
• 新增 `settings.debug_snapshot()`，供 health/debug/tracing 统一使用。
主要风险
• 环境变量名变动会导致线上配置失效，因此第一阶段必须保留旧变量兼容映射。
• 过早清理实验开关会让行为变化，建议先搬迁、后收敛。
• 多个模块直接 import 旧 settings 字段，改动需要配套全局搜索和 smoke test。
验收标准
• 旧 import 路径仍可运行，关键接口不报错。
• 能输出一份当前生效配置摘要，至少包含模型/RAG/存储/观测四类。
• 新增配置时开发者能明确知道该放在哪个子模块。
• 删除旧 config.py 中 70% 以上的真实字段定义，只保留兼容层与极少数公共入口。

四、模块二：拆分 core/graph/sub_graphs/diagnosis.py
当前痛点
• 诊断子图同时承载状态同步、快速分诊、科室路由、query rewrite、工具包装、输出组装与 debug snapshot，职责过重。
• 文件过大导致任何一个诊断相关修改都可能影响整条链路，风险定位困难。
• 科室路由、状态字段写回和 guardrail/repair 缺少明确协议，不利于后续 21 科室与专家组收口。
目标目录与职责
模块/文件	职责
core/graph/diagnosis/state_sync.py	输入 state 规范化、默认值、上下游字段同步。
core/graph/diagnosis/router.py	科室分类、top1/topk 结果、置信度与兜底逻辑。
core/graph/diagnosis/rewrite.py	query rewrite、症状提炼、快速诊断前的输入改写。
core/graph/diagnosis/tools.py	工具调用包装、超时、异常标准化、工具 trace。
core/graph/diagnosis/guardrail.py	诊断输出 schema 检查、repair、保护性裁剪。
core/graph/diagnosis/output_builder.py	最终 diagnosis_output、citations、debug snapshot 组装。
core/graph/sub_graphs/diagnosis.py	只保留子图装配、节点连接与向后兼容导出。

拆分步骤
1. 先画出 diagnosis 当前节点图和 state 字段清单，标记哪些字段是输入、内部字段、输出字段。
2. 把所有 state 读写集中到 `state_sync.py` 和 `contracts.py`，至少定义 department、departments、route_confidence、diagnosis_output、citations 等核心字段。
3. 把 `_run_tool_with_timeout`、quick_triage 工具、异常包装搬到 `tools.py`；统一工具调用返回结构，避免不同调用方各自判断。
4. 把科室路由从主文件中剥离到 `router.py`，并且预留从 registry 读取科室集的入口，不再把候选科室写死在主流程里。
5. 将 diagnosis.py 变薄，只负责 graph 组装和旧函数 re-export，确保第一阶段功能等价。
6. 最后补充 contract tests：给定输入 state，必须得到稳定的核心字段与输出结构。
关键接口与保留兼容
• 新增 `DiagnosisStateContract` 或等价的数据约定，统一字段名与含义。
• 保留旧的子图构造函数签名，避免 ChatService/主工作流一次性联动修改。
• 为 router 暴露 `route_case(case_context) -> RouteDecision`，给专家组与后续 triage 共用。
主要风险
• state 字段名不统一是最大风险，拆分前必须先定协议，否则只是在不同文件间搬乱。
• 路由和 rewrite 分离后，顺序变化可能影响模型输出；第一阶段必须保持调用顺序不变。
• 工具异常标准化后，旧分支可能依赖历史异常字符串，需要有兼容封装。
验收标准
• diagnosis.py 只保留图装配与极少量 glue code，文件体积显著下降。
• 所有诊断相关 state 字段都能在 contracts 中找到定义。
• 路由、rewrite、guardrail、输出组装可以单独测试，不必跑整张图。
• 后续扩科室时，只需要改 router/registry，而不是改整个 diagnosis 主文件。

五、模块三：拆分 rag/retriever.py
当前痛点
• 检索模块聚合了召回、重排、缓存、意图分析、策略切换等多类逻辑，已经成为 RAG 总控中心。
• 大量策略堆叠后，线上一次检索究竟走了什么路径不够清晰，难排障、难做回归分析。
• 不同召回方式和缓存验证逻辑交织，后续再加 GraphRAG、SQL 预过滤或 pure mode 优化会继续膨胀。
目标目录与职责
模块/文件	职责
rag/contracts.py	统一 QueryContext、RecallHit、RerankResult、PipelineTrace 等结构。
rag/query_normalizer.py	问题清洗、症状抽取、重写前标准化。
rag/router.py	判断 pure / hybrid / cached / sql-prefilter / graph 等策略路径。
rag/recallers/vector.py	向量召回。
rag/recallers/bm25.py	词法召回。
rag/recallers/sql_prefilter.py	结构化预过滤。
rag/rerank.py	重排序与阈值判断。
rag/cache.py	语义缓存、验证与回退。
rag/pipeline.py	只负责编排阶段顺序，输出 trace。
rag/retriever.py	保留 MedicalRetriever façade，对外兼容。

拆分步骤
1. 先从现有 retriever 中提取阶段边界：normalize、route、recall、merge、rerank、verify、assemble。
2. 定义统一的 trace 结构，要求每次检索至少记录命中的策略路径、召回源、重排阈值、cache 命中与否。
3. 把召回器先独立成 `recallers/` 子目录；第一阶段不要修改召回算法，只搬迁代码和依赖。
4. 把 cache、rerank、router 抽出后，在 `pipeline.py` 中按原顺序串起来，MedicalRetriever 仅作为 façade 调用 pipeline。
5. 第二轮再收紧配置入口：各阶段只接收自己需要的配置，而不是整个 settings 大对象。
6. 对关键 query 建立 snapshot case，确保拆分前后召回路径和主要结果不出现大偏移。
关键接口与保留兼容
• 保留 `MedicalRetriever.retrieve(...)` 这样的外部调用入口，内部改调 pipeline。
• 定义 `PipelineTrace`，供 Langfuse/日志/评测统一消费。
• 定义 `RecallHit` / `SearchPlan` 等协议，避免不同召回器返回不同字段。
主要风险
• RAG 链路最容易因为隐式顺序变化导致效果漂移，所以先拆结构、后调策略。
• trace 加得不对会引入性能开销，建议只记录关键信息并支持采样。
• 缓存验证与重排阈值之间可能存在历史耦合，拆时要先梳理优先级。
验收标准
• 新增一种召回方式时，只需要加 recaller 与 router 配置，不必改一个超大文件。
• 每次检索都能产出结构化 trace，便于日志、调试和评估系统复用。
• MedicalRetriever 仍可被旧调用方直接使用，保证业务代码改动最小。
• 针对固定测试集，拆分前后主要召回质量和路径差异在可控范围内。

六、模块四：拆分 chat endpoint + chat_service
当前痛点
• 聊天链路同时承担 HTTP 接入、SSE 输出、LangGraph 事件适配、用户可见状态文案与双轨流式节奏控制。
• 当前前端体验和内部节点名绑定较深，改图或改节点名容易影响用户侧状态展示。
• 接口层和服务层职责交叉，后续要加新客户端、新协议或灰度实验会越来越难。
目标目录与职责
模块/文件	职责
api/v1/endpoints/chat_stream.py	请求校验、鉴权、路由、StreamingResponse 封装。
services/chat_orchestrator.py	组织 graph 调用、会话上下文、错误边界。
services/chat_event_adapter.py	把 LangGraph event 转成统一内部事件 schema。
services/chat_stream_renderer.py	把统一事件 schema 渲染成 SSE 数据帧。
services/chat_ux_policy.py	用户可见状态文案、阶段提示、双轨输出策略。
services/chat_contracts.py	请求、响应、事件类型、done/error 约定。
api/v1/endpoints/chat.py	兼容旧路由或转发到新实现。

拆分步骤
1. 先定义统一事件 schema：event_type、phase、node_id、display_text、payload、severity、terminal。
2. 把现有 chat_service 中基于节点名判断的展示逻辑抽到 `chat_ux_policy.py`，不要让 orchestrator 直接拼文案。
3. 把 SSE 文本拼装从业务逻辑中拿出来，独立到 `chat_stream_renderer.py`，以后 WebSocket 或批量回放都能复用事件 schema。
4. endpoint 层只保留 request parsing、鉴权、stream response 与 request id 注入；其余都下沉到 orchestrator。
5. 通过兼容层保留旧 API path 和旧字段名；第一阶段不要同时改前端协议。
6. 补充 SSE contract tests，至少覆盖 status、thought、token、error、done 五类事件的顺序与格式。
关键接口与保留兼容
• 保留现有 `/chat/stream` 或等价路径与主要请求体格式。
• 定义 `ChatEvent` / `ChatChunk` 等协议，前端与后端共同遵守。
• 通过 `ChatUXPolicy` 统一管理对用户可见的状态文案和阶段节奏。
主要风险
• 前端如果依赖某些隐含事件顺序，拆分后容易暴露问题，因此 contract test 非常关键。
• 双轨输出策略与 UX 文案耦合较深，抽离时要避免漏掉 fast track / expert crew 的节奏控制。
• 错误事件、done 事件和取消场景必须统一，不然最容易在流式协议上出事故。
验收标准
• endpoint 只做 HTTP/SSE，服务层只做 orchestration，事件适配与渲染单独存在。
• LangGraph 节点改名时，不需要同步修改一堆用户文案分支。
• SSE 事件格式形成正式协议，并有测试覆盖。
• 未来新增 WebSocket、回放模式或调试 UI 时，可以直接复用统一事件 schema。

七、跨模块配套改造（四块一起做时必须同步推进）
1. 统一 contracts：至少把配置快照、DiagnosisState、RAG Trace、ChatEvent 这四套协议固定下来。
2. 加 request-level trace：一次请求结束后能看到本次模型、路由、检索、专家、fallback 和耗时摘要。
3. 建立三层测试：单元测试（子模块）、contract test（事件/状态/trace 协议）、smoke test（关键问答链路）。
4. 保留兼容层：旧 import、旧 endpoint、旧字段至少保留一个版本周期，避免前后端和其他模块同时爆炸。
5. 每次拆分都要加 feature flag 或灰度入口，确保出问题能快速回退到旧实现。

十一、最终建议
现在不要尝试同时重写专家组、科室体系、治理链路。第一阶段只解决这四块：配置、诊断子图、检索总控、聊天流式桥接。
只要这四块拆开，系统的复杂度就会明显下降，后续再处理 21 科室动态接入、专家组角色化、医疗规则层与演进评测，成本会低很多。
执行原则只有一句：先立协议，后拆实现；先保兼容，再做收敛。

十二、项目定制修订版（将适配度从 8.5/10 提升到 10/10）

下面这版是“按当前仓库真实结构”修订后的升级三，不再按通用模板描述。

A. 现状与修订点（必须按真实代码落地）
1) chat 主链路不在 chat_service，而在 endpoint：
- 当前真实主链路：`backend/app/api/v1/endpoints/chat.py` 的 `event_generator`。
- `backend/app/services/chat_service.py` 仅保留为兼容层，不作为主拆分目标。
- 修订动作：模块四改为“chat.py-first 拆分”，chat_service 只做 legacy wrapper。

2) diagnosis 协议已有 TypedDict 基座：
- 当前已有 `backend/app/domain/states/sub_states.py::DiagnosisState`。
- 修订动作：先冻结该契约，再新增 `core/graph/diagnosis/contracts.py` 做运行时字段约束与校验辅助，不推翻现有类型。

3) retriever 外部入口必须稳定：
- 当前外部依赖 `MedicalRetriever.search_rag30(...)` 和 `get_retriever()`。
- 修订动作：`rag/pipeline.py` 只做内部编排，`retriever.py` 保留 façade 与原签名，避免上层 diagnosis 联动爆炸。

4) config 存在初始化副作用：
- `settings = Settings()` 在初始化里包含 key 轮转、环境变量导出、副作用日志。
- 修订动作：先拆字段，再抽行为 helper；`settings` 单例入口与副作用时序保持不变。

B. 四块拆分的项目专属目标路径（以当前代码为准）
1) 配置层（保持入口不变）
- 保留：`from app.core.config import settings`
- 新增：
  - `backend/app/core/settings/base.py`
  - `backend/app/core/settings/storage.py`
  - `backend/app/core/settings/model.py`
  - `backend/app/core/settings/rag.py`
  - `backend/app/core/settings/observability.py`
  - `backend/app/core/settings/feature_flags.py`
  - `backend/app/core/settings/runtime_side_effects.py`（承接 key/export_env 等行为）
- 兼容层：`backend/app/core/config.py` 最终只保留 façade、别名映射与单例装配。

2) diagnosis 子图（薄化主文件）
- 当前目标文件：`backend/app/core/graph/sub_graphs/diagnosis.py`（超大文件）
- 新增：
  - `backend/app/core/graph/diagnosis/contracts.py`
